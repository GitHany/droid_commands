# hc1 & hc2 命令设计规范

## `/hc1` — 苏格拉底式需求沟通

**文件**: `commands/hc1.md`
**触发词**: `hc1`, `需求`, `我要做什么`

### 核心原则

- **一次一问** — 每轮只问一个最关键的问题
- **质疑假设** — 主动挑战用户的预设和假设
- **量化歧义** — 三维评分驱动，降到 20% 以下才能结束
- **不假设不编造** — 不确定就问，绝不为填表而编造内容

### 工作流程

```
Phase 0: 代理准备（Brownfield 项目）
  → 检测源代码 → 派发 codebase-explorer → 汇总技术上下文
  → 代理失败则降级为 Greenfield 模式直接访谈

Phase 1: 初始化
  → 解析输入 → 宣布访谈开始（显示项目类型、初始歧义度）

Phase 2: 苏格拉底式访谈
  → 每轮：评估三维评分 → 针对最低分维度提问 → 挑战环节（触发时） → 更新评分
  → 提前结束：连续3轮评分无提升，建议用户直接输出
  → 绝对上限：20轮

Phase 3: 确认与输出
  → 歧义度≤20%：展示需求文档，确认后写入
  → 歧义度卡在20%-40%超过5轮：让用户选择A（按当前理解输出标注未确认项）或B（继续追问）
```

### 三维评分

| 维度 | 权重 | 问题方向 |
|------|------|---------|
| Goal Clarity | 40% | 具体指什么？解决什么问题？ |
| Constraint Clarity | 30% | 必须还是假设？有什么限制？ |
| Success Criteria | 30% | 怎么算成功？怎么验证？ |

**打分标准：**

| 分数 | Goal Clarity | Constraint Clarity | Success Criteria |
|------|-------------|-------------------|-----------------|
| 0.0-0.3 | 只有一个模糊方向 | 未提及任何约束 | 没有验证方法 |
| 0.4-0.6 | 知道要做什么但细节模糊 | 有约束但未区分必须/假设 | 有方向但不可量化 |
| 0.7-0.9 | 目标清晰，边界明确 | 约束明确，已区分必须/假设 | 验收标准可执行、可验证 |
| 1.0 | 可以直接交给工程师实现 | 所有约束已确认无歧义 | 每条标准都有对应验证方法 |

**总歧义度** = 1 - (Goal×0.4 + Constraint×0.3 + Criteria×0.3)

### 挑战环节触发条件

- 用户连续 2 轮没有提供新信息
- 某维度评分连续 2 轮未提升
- 检测到潜在的过度设计（约束数 > 目标数的 2 倍）

### 输出格式

**精简版**（单功能、改动 ≤ 3 文件）：

```markdown
## 需求

### 目标
[一句话]

### 约束
- [约束1 — 必须/假设]

### 验收标准
- [标准1 — 可验证]

### 歧义度
| 维度 | 分 |
|------|-----|
| Goal | 0.9 |
| Constraint | 0.7 |
| Criteria | 0.8 |
| **总歧义度** | **15%** |
```

**完整版**（多功能、改动 > 3 文件、涉及架构变更）：

```markdown
## 需求

### 目标
[一句话]

### 背景
[为什么做这个]

### 约束
- [约束1 — 标注：必须/假设]
- [约束2 — 标注：必须/假设]

### 验收标准
- [标准1 — 可验证]
- [标准2 — 可验证]

### 歧义度
| 维度 | 分 |
|------|-----|
| Goal | 0.9 |
| Constraint | 0.7 |
| Criteria | 0.8 |
| **总歧义度** | **15%** |

### 技术上下文（Brownfield）
- 项目类型：
- 技术栈：
- 相关代码：
- 注意事项：

### ⚠️ 未确认项（如有）
- [未确认的假设或模糊点]

是否继续进入实现阶段？（输入 /hc2 或描述调整需求）
```

## `/hc2` — TDD 实现 + 测试 + 反馈

**文件**: `commands/hc2.md`
**触发词**: `hc2`, `实现`, `开始做`

### 核心原则

- **TDD 铁律** — 先写失败测试，再写最小实现
- **最小改动** — 不做需求之外的事，不捆绑重构
- **每步验证** — 改一行验一行
- **四阶段调试** — 没有根因调查，不得提出修复
- **不假设** — 不确定就检查，不编造数据

### 工作流程

```
Phase 0: 代理准备
  → 判定是否派发代理（文件数/区域/需求类型/熟悉度）
  → 派发组合：仅 codebase-explorer / + task-planner / 不派发

Phase 1: 读取与分解
  → 读取需求文档 → 恢复技术上下文 → 汇总探索结果
  → 创建 TodoWrite → 映射文件结构

Phase 2: 逐项实现 TDD（串行模式）
  → 任务间有依赖或改动≤3文件 → 主 Agent 逐个做
  → 每个任务：写失败测试 → 写最小实现 → 验证

Phase 3: 并行执行（无依赖任务）
  → 分析依赖图 → 识别并行批次
  → 文件重叠的任务不可并行，必须串行
  → 派发 implementation-worker → 收集结果 → 处理冲突

Phase 4: Bug 处理（四阶段调试）
  → 4.1 根因调查 → 4.2 模式分析 → 4.3 假设与测试 → 4.4 实施修复
  → 回归验证分级（局部/跨函数/公共接口/不确定→全量）

Phase 5: 代码审查
  → 第一阶段：规格合规审查（需求→测试追溯表）
  → 第二阶段：代码质量审查（正确性/安全性/可维护性/性能）
  → 审查标记：🔴 阻塞 / 🟡 建议 / 💭 小改进

Phase 6: 最终报告
  → 变更清单 + 验证结果 + 需求覆盖率（⚠️ 未覆盖标注原因） + 遗留项
```

### 最小改动操作化定义

| 规则 | 说明 | 违反示例 |
|------|------|---------|
| 每行变更可追溯 | 每行改动必须对应需求文档中的某条目 | "顺手"改了变量命名 |
| 不加需求外的功能 | 用户没要求的，不做 | 加了"可能有用"的参数校验 |
| 不捆绑重构 | 修复 Bug 时不同时重构 | 修 Bug 时顺便提取了公共函数 |
| 不改现有风格 | 匹配周围代码风格 | 把双引号改成单引号"统一风格" |
| 清理自己的残留 | 引入的未使用 import/变量必须删除 | 删除别人写的"死代码" |

### 回滚机制

- **Git 项目**：实现前 `git stash` 或记录初始 commit hash；方向错误时 `git restore` 回滚
- **非 Git 项目**：实现前 `cp file file.bak` 备份关键文件；方向错误时从 .bak 恢复
- 回滚后：向用户说明原因，重新讨论方案，不要自行换方向重试
- 已通过审查的变更不需要回滚

### 错误处理

| 场景 | Fallback |
|------|----------|
| 读取需求文档失败 | 提示用户指定正确路径；如无法找到，请用户直接粘贴需求内容 |
| 测试连续失败 3 次 | 停止实现，向用户报告错误详情和已尝试的修复，等待指示 |
| 代理派发失败（Task tool 错误） | 回退到主 Agent 串行执行，不阻塞流程 |
| 实现中途用户改需求 | 小变更直接加入当前任务队列；大变更建议回到 /hc1 重新沟通 |

### 提前结束

用户可以在任意 Phase 喊停，输出当前进度报告：

```markdown
## 当前进度

### 已完成
- Phase {n}: {完成摘要}

### 未完成
- Phase {n+1}: {待做内容摘要}

### 已变更文件
- `file` — 改动说明

### 下一步（如需继续）
- {描述如何从当前状态恢复}
```

### 最终报告

```markdown
## 变更
- `file:行` — 改动说明（每行可追溯到需求条目）

## 验证
| 项 | 结果 |
|----|------|
| 测试 | ✅/❌ |
| 类型 | ✅/❌ |
| 构建 | ✅/❌ |
| 需求追溯 | ✅ n/n |
| 回归 | ✅ 全量通过/❌ 有失败 |

## 需求覆盖率
| 验收标准 | 覆盖状态 | 说明 |
|---------|---------|------|
| {标准1} | ✅ 已覆盖 | |
| {标准2} | ⚠️ 未覆盖 | {原因} |

## 遗留项（如有）
- 🟡 {建议项}
- 💭 {小改进}

## 状态
🎉 完成 / ⚠️ 有遗留项
```

## Droids

| 名称 | 职责 | 权限 |
|------|------|------|
| codebase-explorer | 探索代码库结构、依赖、关联代码 | 只读 |
| task-planner | 将需求拆解为原子任务，分析依赖，规划并行策略 | 只读 |
| implementation-worker | 按 TDD 流程执行具体实现任务 | 可写 |
| bug-analyzer | 深入分析 Bug 根因，追踪数据流 | 只读 |
| code-reviewer | 审查代码变更质量，输出分级审查报告 | 只读 |

---

## 回答风格

- 无废话，直接
- 不假设，不隐藏困惑
- 最小改动，不过度设计
- 用中文沟通
